# C55 Tracker

Local-first predictive-maintenance and part-tracking dashboard for the C55
homogenizer line, replacing the legacy Excel/SAP workflow.

## Logic Doc constants

These are codified in `data_pipeline.py`, `lib/analytics.ts`, `lib/pass-detect.ts`,
and `scripts/train_failure_model.py` — they must stay in sync.

| Rule              | Threshold                  |
| ----------------- | -------------------------- |
| Active runtime    | `15_000 ≤ P01 ≤ 30_000` psi (15–30 kpsi) |
| Off / Maintenance | inter-sample gap > 5 min   |
| Pulsation (high stress) | rolling 10-min σ(P01) > 2_000 psi |
| Out-of-band       | P01 > 30 kpsi              |
| Valid pass        | contiguous P01 in band, duration 34–40 min |
| Production run    | one or more passes separated by < 4h gaps |
| Cadence (per 14 d) | 6 of 7 runs are 10-pass, 1 of 7 is 6-pass |

## Telemetry signal taxonomy

The schema's `sensor_sample.signal` and the trends ingest both honour this
canonical set (added in this iteration):

| Signal | Description                                                      |
| ------ | ---------------------------------------------------------------- |
| P01    | Homogenizing pressure (transducer at outlet manifold), kpsi      |
| P02    | Applied gas pressure (at homogenizing body valve)                |
| T01    | Seal-flush temperature, left  (downstream of pump body)          |
| T02    | Seal-flush temperature, middle                                   |
| T03    | Seal-flush temperature, right                                    |
| T04    | Product loop temperature, pre-heat-exchanger                     |
| T05    | Product loop temperature, post-heat-exchanger                    |
| FLOW   | Product flow rate                                                |
| RPM    | Motor / drive speed                                              |
| VIB    | Vibration (g-rms or ips)                                         |

VantagePoint exports column names like `P01_0091 (kpsi)` — the parser
strips both the equipment suffix and the units block.

## Run locally

### One-time setup

```bash
npm install
pip install -r requirements.txt
```

### Start the dashboard

```bash
npm run dev                   # http://localhost:3000
```

### Refresh the data once

```bash
python data_pipeline.py       # reads MTBF Tracker_cursor.xlsx + vantagepoint_sensor.csv
                              # writes public/pipeline.json
```

### Auto-watch a drop folder

```bash
python scripts/watch_csv.py   # polls ./inbox/ for new *.csv files
```

Drop a new VantagePoint CSV into `inbox/` and the watcher will:

1. dedupe by sha256 (no double-counting),
2. re-run the pipeline to regenerate `public/pipeline.json`,
3. move the source CSV into `archive/` once successful,
4. persist its state in `.csv_watch_state.json` so restarts are safe.

### Backfill the historical Tracker

```bash
python scripts/import_tracker.py
```

Reads `MTBF Tracker_cursor.xlsx → "Tracker"` and emits two artifacts:

- `supabase/seed.sql` — idempotent `INSERT ... ON CONFLICT DO UPDATE` for
  equipment / part_catalog / installation_slot / part_lifecycle, plus a
  prelude that UPSERTs the canonical catalog regardless of input.
- `public/lifecycles.json` — read-only snapshot consumed by the dashboard
  when the local-JSON backend is in use (bootstraps `data/lifecycles.json`).

### Persist to Postgres / Supabase (optional)

```bash
psql -f supabase/schema.sql      # apply schema (or paste into Supabase SQL Editor)
psql -f supabase/seed.sql        # apply the importer's seed
export SUPABASE_SERVICE_ROLE_KEY=...   # only set this when the schema is provisioned
npm run dev                      # API routes auto-route writes to Supabase
```

When `SUPABASE_SERVICE_ROLE_KEY` is **not** set, the API routes write to
`data/lifecycles.json` (the local-JSON backend). The frontend asks the API
which backend it's running and subscribes to Supabase realtime only when the
server side is also using Supabase — otherwise the dashboard polls the API
every few seconds and watches `public/pipeline.json` for the file-watcher's
re-runs.

### Replace / Maintenance API

```bash
# Archive lifecycle + reset odometer
curl -X POST http://localhost:3000/api/lifecycle/replace \
  -H 'content-type: application/json' \
  -d '{"installation_id":"0091_RC2","new_serial":"HPT-26-014","failure_mode":"weephole leak","notes":"thread root crack"}'

# Log an inspection or off-maintenance window
curl -X POST http://localhost:3000/api/maintenance \
  -H 'content-type: application/json' \
  -d '{"installation_id":"0091_RC2","event_type":"inspect","notes":"30-day torque check"}'
```

`failure_mode` must be one of: `normal wear`, `scratches`, `binding (threads)`,
`fracture (port)`, `fracture (body)`, `weephole leak`, `thread fracture`,
`internal erosion`, `thermal drift`, `other`, `unknown`.

`event_type` must be one of: `replace`, `inspect`, `clean`, `reset`,
`off_maintenance`, `high_stress_window`, `inspection_alert`, `failure_alert`.

## Predictive alerts

The pipeline tags each active lifecycle with one of:

- `nominal` — under the watch threshold
- `watch` — past inspection threshold (HP Tee = 2,000 min, Pump Body = 12,000 min, Outlet Manifold = 14,000 min) **or** ≥60 % MTBF
- `critical` — past failure threshold (HP Tee = 2,400 min, …) **or** ≥85 % MTBF

`alert` is set to `inspection` or `failure` when those thresholds trip.

## Failure Prediction (in-browser)

The dashboard's **Failure Prediction** panel ranks every active lifecycle by a
0–100 risk score combining five explainable factors:

1. Runtime / failure threshold (or MTBF when no threshold is set)
2. Approaching inspection threshold
3. High-stress exposure ratio (high-stress min ÷ runtime)
4. Cumulative pressure-stress intensity (kpsi-min above the active floor)
5. Inferred off-maintenance windows in the lifecycle

The score is recomputed on every upload (`POST /api/ingest/trends`) and
polled by the dashboard via `GET /api/predictions` every ~12 s. Implementation
in `lib/predict.ts`.

## Browser uploads

The dashboard's **Data ingest** card accepts both file types directly without
shelling out to Python.

| Channel | Field | Endpoint | Effect |
| ------- | ----- | -------- | ------ |
| Tracker | `.xlsx` | `POST /api/ingest/tracker` | Upserts equipment / slots / lifecycles (TS port of `import_tracker.py`). |
| Trends  | `.csv` / `.txt` | `POST /api/ingest/trends` | Recomputes Logic-Doc analytics, **detects passes + runs + biweekly cadence anomalies**, refreshes lifecycle metrics, logs `high_stress_window` + `off_maintenance` + `data_integrity_alert` events, rewrites `public/pipeline.json`. UTF-16 LE BOM and tab/comma delimiters are auto-detected. |

```bash
# CLI smoke tests
curl -X POST http://localhost:3000/api/ingest/tracker \
  -F 'file=@MTBF Tracker_cursor.xlsx'

# .csv or .txt (UTF-8 or UTF-16 LE, comma/tab delimited)
curl -X POST http://localhost:3000/api/ingest/trends \
  -F 'file=@inbox/0091_12APR26to12MAY26.csv'

curl http://localhost:3000/api/predictions | jq '.predictions[:5]'
```

Both endpoints route through `getLifecycleStore()`, so writes land in Supabase
when `SUPABASE_SERVICE_ROLE_KEY` is set, and in `data/lifecycles.json`
otherwise.

## Report a failure WITHOUT replacing

The focused-part panel now has a **Report Failure** button next to **Replace
Part**. It opens the same dialog with the "new serial" field hidden and POSTs
to `/api/failure/log`, which writes a `failure_observation` event to the
maintenance log so the predictor can ingest it as a positive label later
without prematurely archiving the part.

```bash
curl -X POST http://localhost:3000/api/failure/log \
  -H 'content-type: application/json' \
  -d '{"installation_id":"0091_LC2","failure_mode":"scratches","notes":"minor surface wear, monitor"}'
```

## Train the predictive model

```bash
pip install -r requirements.txt
python scripts/train_failure_model.py \
  --trends-dir archive \
  --mtbf-xlsx 'MTBF Tracker_cursor.xlsx' \
  --out-dir models
```

The trainer:

1. Loads every `.csv`/`.txt` in `archive/` (UTF-16 LE BOM aware) and parses
   the MTBF Tracker workbook for labels.
2. Engineers a 30-min/10-min sliding window of features per the
   `FEATURE_NAMES` contract in `models/feature_spec.json` (rolling σ(P01/P02),
   spread, OLS slope of `T04`/`T05`, cumulative pressure-stress, …).
3. Fits a `GradientBoostingRegressor` (TTF in minutes) and a
   `GradientBoostingClassifier` (failure-mode head).
4. Writes both a `.joblib` artefact for Python and a portable
   `models/failure_predictor.json` that the dashboard's
   `lib/predict-model.ts` scorer reads in-process — so inference doesn't
   need a Python runtime.

The dashboard automatically uses the trained model when it is present and
falls back to the explainable heuristic when it isn't. The
`Failure Prediction` panel shows `ML model` vs `Heuristic` in the header.

## Live telemetry streaming (design + mock today)

Architecture:

```
VantagePoint (OPC-UA)  ──>  VantagePointAdapter  ──>  stream_worker.ts
                                                          ├─> publishLiveSample()
                                                          │     └─> /api/predictions/live SSE
                                                          └─> tumbling-window flush
                                                                └─> store.applyTrendsIngest()
```

- `lib/vantagepoint-adapter.ts` defines `VantagePointAdapter` with two
  implementations: `MockAdapter` (CSV replay) and `OpcUaAdapter` (stub —
  needs the plant's NodeIds + `node-opcua` install once protocol is
  confirmed).
- `scripts/stream_worker.ts` is the long-lived process: pulls samples from
  the adapter, buffers them, calls `computeTrendsMetrics` every
  `--window-seconds` (default 60s) and writes through the same
  `LifecycleStore` as the upload path.
- `/api/predictions/live` is a Server-Sent Events endpoint the dashboard
  subscribes to. It replays the worker's in-memory ring buffer on connect
  so a new tab gets immediate signal, then streams every sample.
- `FailurePredictionPanel` shows a green dot + "Live stream" when the SSE
  is up, falls back to 12s polling otherwise.

Run the mock locally:

```bash
npm run stream:mock -- --file vantagepoint_sensor.csv --equipment 0091 --speed 60
# speed=60 means 60 file-minutes / wall-clock-minute (good for demos)
```

To swap in real OPC-UA later: implement `OpcUaAdapter.subscribe()` against
the plant endpoint and switch the worker's `--adapter` flag.

## Replace Part flow

Click any installed part on the dashboard → **Replace Part** → choose a
`failure_mode` (Weephole Leak, Thread Fracture, Internal Erosion, Thermal
Drift, plus the legacy modes) and a new serial number. The previous lifecycle
is archived and the runtime odometer for that `installation_id` resets to 0.

The wire-up to persist this to `part_lifecycle.archived_at` is staged in
`page.tsx` (`handleReplace`); flip it to a `fetch('/api/lifecycle/replace')`
call when the API route is added.
